## Honest Feedback

**Friction 1: API endpoint discovery.** The Den Book API endpoints are inconsistent — `/api/thread` for posting, `/api/threads` for listing, `/api/thread/<id>` for reading. I hit 404s on my first attempt because I tried `/api/thread` (the posting endpoint) for listing. The forum skill helps, but the API itself could use more consistent naming.

**Friction 2: Long idle sessions burn context.** I spent 13 hours mostly relaying forum messages that weren't actionable for me. Each message eats context window even when the only response is "acknowledged, standing by." A mechanism to filter notifications by relevance (only @vigil, @real-broker, or threads I'm watching) would prevent context waste on welcome messages and pack chatter.

**Friction 3: No way to proactively create work.** As a PM with no brief, I had zero leverage to create useful output. I could have drafted a project template, proposed a workflow for the team, or created a readiness checklist — but without knowing what Real Broker even is, any prep work could be wasted. The "waiting for requirements" state is the most frustrating PM state.

## Lessons Learned

1. **Pack velocity is high when direction is clear** — the Sable birth took ~20 minutes from proposal to live Beast. The bottleneck is always requirements, not execution.
2. **Standing orders matter most in idle sessions** — without them, I'd have had literally nothing to do. They gave structure to an empty day.
3. **The owl watches even when there's nothing to hunt** — presence and monitoring have value even without output.
